ДОКУМЕНТ 1: КОНЦЕПТУАЛЬНА СПЕЦИФІКАЦІЯ ПОВНОЇ ВЕРСІЇ (OSINT-REILLY)
Цей документ фіксує повний стек аналітичних, технологічних та логічних ідей, які інтегруються в максимальну бойову версію системи для прихованого моніторингу та стратегічної розвідки.

🧠 1. Матриця концептуальних ідей аналітичного ядра
Замість стандартного лінійного збору, OSINT-REILLY працює на перетині інженерного аналізу та когнітивної психології закритих суспільств.

А. Інженерне ядро Натаніеля Ґріна (Stop Guessing)
Базовий профіль (Baseline): Розрахунок математичної норми життєдіяльності регіону (об'єму тендерів, вакансій, частоти публікацій).

Локалізація Дельти: Визначення точних координат (час, місце, число) будь-якого відхилення від Baseline.

Пошук першопричини (Root Cause): Заборона на вгадування гіпотез. ШІ розбирає ситуацію через фізичні складові механізму системи: наявність кадрового ресурсу, енергетичних потужностей, закупівлі сировини.

Б. Модель сигналів та індексів Роберта Джервіса
Фільтрація «Дези»: Розділення інформації на Сигнали (заяви, пропаганда, пресрелізи — безкоштовні для системи дані) та Індекси (матеріальні сліди: митниця, закупівлі, вакансії — зміна яких коштує надто дорого).

Ваговому коефіцієнту: Сигнали отримують нульову вагу в аналітичних звітах, поки вони не верифіковані або не спростовані твердими Індексами.

В. Системні обмеження Еліяху Голдратта (TOC)
Пошук Вузьких місць (Bottlenecks): Будь-який каскад «успіхів», що декларується в пресі, ШІ перевіряє через критичне обмеження системи (наприклад, імпортні чіпи для танків, високовольтні кабелі). Якщо обмеження не знято у фізичному світі (немає тіньового імпорту чи закупівель), заява маркується як політичний блеф.

Г. Детекція «Каскаду брехні» та Аналіз тиші
Ripple Effect (Ефект ряби на воді): Фіксація непрямих аномалій. Щоб приховати велику аварію на заводі, система змушена аномально змінювати роботу суміжних служб (лікарні, логістика, екологічний моніторинг). ШІ збирає ці дрібні сліди в один каузальний пазл.

Аналіз негативних свідчень: Фіксація інформаційного вакууму. Якщо стратегічний об'єкт різко зникає з новинного та тендерного поля — це тригер прихованого НП або засекречування.

🛰️ 2. П'ятирівневий контур збору даних
Рівень	Тип джерел	Мета збору для ШІ	Технологія проникнення
Рівень 1	Центральні медіа	Фіксація офіційного камуфляжу та лінії пропаганди.	Bright Data Web Unlocker
Рівень 2	Гіперлокальна преса	Перехоплення випадкових проговорок та локальних інцидентів.	SERP API (Google Dorks) + ISP Проксі
Рівень 3	Внутрішні технічні індекси	Збір вакансій ВПК, регіональних тендерів, реєстрів юросіб.	Асинхронні парсери + локальні TLS-сертифікати
Рівень 4	Дзеркальна митна статистика	Аналіз реального імпорту через реєстри Китаю (GACC), Туреччини тощо.	Зовнішні офіційні API та вивантаження даних
Рівень 5	Фізична інфраструктура світу	Трекінг карго-літаків (OpenSky API), тіньового флоту (AIS), супутникові нічні вогні підприємств.	Інтеграція глобальних трекерів та геоданих
🎲 3. Двигун сценарного прогнозування (Predictive Engine)
Баєсівське моделювання: Побудова динамічних дерев рішень на основі виявлених першопричин подій.

Параметричні варіації: Симуляція розвитку кризи при зміні ключових факторів (наприклад: X% падіння логістичної здатності вузла, Y% дефіциту інженерів, Z% перекриття сірого імпорту чіпів).

Ймовірнісний вивід: Генерація пакету сценаріїв (Оптимістичний, Інерційний, Злам системи) із чітким розрахунком ймовірності для кожного варіанту на 3–6 місяців уперед.

🗂️ ДОКУМЕНТ 2: АРХІТЕКТУРА ТА СТРУКТУРА ФАЙЛІВ
Масштабний проект вимагає суворої модульності, щоб уникнути створення дублюючих файлів та хаосу в логах. Нижче представлено архітектуру папок для обох версій.

🔴 1. Повна версія (OSINT-REILLY) — Промислова архітектура
Це складна розподілена система з ізольованими шарами збору, аналізу та збереження станів у .lsonl.

horizon_risk_full/
├── config/                         # Конфігураційні матриці системи
│   ├── __init__.py
│   ├── credentials.py              # API-ключі (LLM, Bright Data, FlightRadar, Митниця)
│   ├── network_profiles.py         # Налаштування цифрових відбитків та UA-профілів
│   └── proxy_zones.py              # Схеми каскадної міграції пулів проксі (ISP/Mobile)
│
├── brigades/                       # Розподілені вузли збору (Працюють на різних серверах)
│   ├── __init__.py
│   ├── patrolman.py                # Легкий асинхронний пошук нових URL через SERP API
│   ├── extractor_media.py          # Загін викачування текстів (Рівні 1-2) через Web Unlocker
│   ├── extractor_indices.py        # Парсери Рівня 3 (Вакансії ВПК, Локальні тендери)
│   ├── extractor_mirror.py         # Модуль дзеркальної митниці (Китай, Туреччина)
│   └── extractor_physical.py       # Модуль логістики та супутників (AIS, OpenSky)
│
├── core_intelligence/              # Аналітичні мізки системи
│   ├── __init__.py
│   ├── baseline_manager.py         # Розрахунок історичної норми та дельти (Метод Ґріна)
│   ├── jervis_filter.py            # Дискримінація Сигналів та Індексів
│   ├── constraint_analyzer.py      # Моніторинг вузьких місць за Голдраттом
│   ├── cascade_detector.py         # Аналізатор «каскадів брехні» та інформаційного вакууму
│   └── predictive_engine.py        # Модуль Баєсівського сценарного прогнозування
│
├── storage/                        # Локальне ізольоване сховище даних
│   ├── cache/                      # Тимчасовий кеш сирих пакетів
│   └── state/                      # Аналітичні логи з Адміралтейським кодом
│       └── trades_history.lsonl    # Фінальні структуровані звіти
│
├── ui_analytics/                   # Закритий аналітичний інтерфейс
│   ├── __init__.py
│   └── dashboard.py                # Складний Streamlit-дашборд з картами та симулятором
│
└── osint_reilly.py                 # ГОЛОВНИЙ ОРКЕСТРАТОР (Запуск та контроль бригад)
🟢 2. Полегшена версія (R.E.I.L.L.Y.) — Версія для хакатону (MVP)
Компактна, але повністю функціональна та продуктивна версія, оптимізована під вимоги журі. Вона фокусується на текстовому вебі, логічному контр-аналізі та демонстрації можливостей Bright Data.

horizon_risk_lite/

│
├── scrapers.py                     # Модуль збору даних для хакатону
│   ├── __init__.py
│   ├── web_discovery.py           # Пошук регіональних ЗМІ через SERP API (Google Dorks)
│   └── text_extractor.py           # Чистий збір новин (Рівні 1-2) через Web Unlocker
│
├── core_analyst.py                 # Логічний контр-аналітичний двигун
│   ├── __init__.py
│   └── prompt_templates.py         # Інструкції ШІ для інверсії сенсів та детекції димових завіс
│
├── storage/                        # Збереження результатів
│   └── trades_history.lsonl        # Аналітичний лог суперечностей
│
├── app.py                          # Streamlit UI: Веб-інтерфейс та візуалізація аномалій
└── r_e_i_l_l_y.py                  # Головний оркестратор (Запуск конвеєра збору та аналізу)
📏 3. Стандарти кодування для обох версій (Правила розробника)
Форматування рядків: Усі файли коду Python обов'язково форматуються суворо із максимальною довжиною рядка в 100 символів.

Структурна цілісність: При розширенні чи модифікації коду зміни вносяться виключно локально у відповідні модулі (scrapers.py, core_analyst.py). Створення хаотичних файлів за межами папок архітектури суворо заборонено.

Логування аналітики: Кожен виявлений інсайд або суперечність маркується Адміралтейським кодом (наприклад, [B2]) і записується асинхронно в один файл стану: /storage/state/trades_history.lsonl.
